# 万能导共创邀请：一起把更多平台接进来

万能导已经开源，也正在从“单个工具”升级成“平台共创生态”。知识平台太多了：飞书、语雀、有道、阿里云、OneNote、为知、印象笔记、Notion、各种博客和企业知识库，靠一个人很难长期覆盖所有导入导出需求。

所以我们希望把万能导做成一个更容易共创的项目：每个平台都可以作为独立签名插件接入。简单平台使用标准 UI，复杂平台可以包含多个导入/导出 Provider，必要时使用沙箱自定义 UI；插件合并后独立发布，用户无需等待主程序更新。

## 谁可以参与

- 你是某个平台的重度用户，知道它怎么导出、导入或迁移。
- 你会一点 Python，愿意把重复操作写成脚本。
- 你在用 AI 编程工具，想通过 vibe coding 接入一个新平台。
- 你不会写代码，但愿意补充教程、截图、失败原因和测试反馈。
- 你已经踩过某个平台的坑，愿意把经验写成排查文档。

不用一开始就做得很完整。一个清楚的教程、一段能跑通的最小脚本、一次真实测试结果，都是有价值的贡献。

## 可以贡献什么

- 新平台导出：把平台内容导出为 Markdown，并尽量保留目录、图片和附件。
- 新平台导入：把本地 Markdown 导入到目标平台。
- 教程型平台：如果平台本身已经支持导出导入，可以先写一篇正确教程。
- 失败排查：整理登录过期、权限不足、图片缺失、接口限流等常见问题。
- 体验优化：让错误提示更清楚，让批量任务更稳定，让报告更容易看懂。

## 新平台怎么接入

万能导提供了三种模板：

```text
providers/_template_standard/
providers/_template_import/
providers/_template_custom/
```

标准导出平台优先复制 `providers/_template_standard/`，标准导入平台优先复制 `providers/_template_import/`，只需要改 `provider.json`、`README.md` 和 `actions.py`。

复杂平台可以复制 `providers/_template_custom/`，先把流程拆成登录、权限检测、读取目录、导出或导入、失败重试等动作。如果标准 UI 不够，也可以在 PR 里说明需要什么专属 UI。

完整说明见：

```text
docs/在线插件开发与发布.md
docs/插件开发指南.md
docs/Provider接入说明.md
```

如果你使用 AI 编程工具，可以直接复制这份通用提示词：

```text
docs/tutorials/provider-ai-prompt.md
```

## 我们更欢迎怎样的 PR

- 平台目录独立，不把特殊逻辑随意塞进主程序。
- README 写清楚使用步骤、登录方式、权限要求、限制和测试结果。
- `provider.json` 如实声明支持能力，不把未完成的功能写成已支持。
- 脚本最后输出 JSON，运行中输出清楚的进度和失败原因。
- 不提交 Cookie、Token、账号密码、私人知识库内容或未脱敏截图。
- 如果参考了第三方项目，请说明来源和许可证。

## 推荐从这些方向开始

- 把你正在用的平台做成教程型 provider。
- 给已有平台补充“失败原因和解决办法”。
- 给导入导出脚本补图片、附件、目录结构、断点续跑或失败重试。
- 新增你常用平台的导入或导出功能，先从最小可用版本开始跑通。

## 开源协议和边界

万能导采用 AGPL-3.0-only 协议开源。提交 PR 默认表示你同意按本项目协议授权贡献内容。

万能导只用于整理你有权限访问和迁移的内容。请不要提交绕过登录、绕过权限控制、破解平台限制或批量获取未授权内容的实现。

如果你愿意一起共创，欢迎从一个小平台、一篇教程、一个失败案例开始。平台越多，万能导越需要大家把真实使用经验带进来。
