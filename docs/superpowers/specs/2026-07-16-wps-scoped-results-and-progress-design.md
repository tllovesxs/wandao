# WPS 专属失败结果与逐文件进度设计

## 背景

当前 `812603d` 将“最新任务结果”容器放到了平台内容之外的全局区域，导致 WPS 导出失败项在主页、任务中心或其他平台页面也可能显示。WPS 后端虽然接收 `--progress-every`，但没有在导出循环中使用它，也没有输出项目已经支持的结构化逐文件进度事件。

## 目标

1. WPS 导出失败结果只属于 `wps-export` 页面，离开 WPS 页面立即隐藏。
2. 主页、任务中心和其他平台页面不显示 WPS 专属失败结果区。
3. WPS 专属结果区移除“查看任务中心”和“打开报告”。
4. 任务中心现有的通用任务历史记录保持不变。
5. WPS 导出按文件输出开始、完成、失败和总进度日志。
6. 不改变 WPS 以外平台的 UI、Provider 或后端行为。

## 方案比较

### 方案 A：回退全局布局，再在 WPS 页面内部重建（采用）

先撤销 `812603d` 中把结果卡移到全局外层的部分，再把专属容器放进 WPS Provider 内容区域。渲染入口必须同时满足当前 Provider 为 `wps-export` 且最近结果属于 WPS。

优点：组件所有权清晰；切换平台天然卸载；不会形成全平台隐式依赖。

缺点：需要重新整理少量 WPS UI 测试和样式。

### 方案 B：保留全局容器，仅通过条件隐藏（不采用）

继续保留外层结果卡，只在非 WPS 页面设置隐藏。

优点：修改较少。

缺点：组件依然属于全局模块，容易再次泄漏到主页或任务中心，不符合“固定在 WPS 模块”的要求。

### 方案 C：新增通用插件扩展槽（不采用）

为所有 Provider 建立可插拔结果面板协议，再由 WPS 注册自己的面板。

优点：长期扩展性强。

缺点：会修改全平台架构和通用契约，超出本次 WPS 范围。

## UI 设计

### 归属和生命周期

- WPS 结果容器只能由 `wps-export` Provider 页面创建。
- 页面切换、返回主页或进入任务中心时，WPS 容器不参与渲染并清空临时展开状态。
- 只显示最近一条 `providerId === "wps-export"` 的导出结果；登录、扫描和清理认证不作为失败结果卡展示。
- 通用任务中心继续读取任务历史，不增加或删除 WPS 特例。

### 卡片内容

- 标题：最近一次 WPS 导出结果。
- 摘要：总数、成功、跳过、失败、耗时。
- 失败项：默认可折叠；部分完成或失败时默认展开。
- 操作可包含：重试失败项、打开输出、复制失败项、复制报告。
- 明确删除：查看任务中心、打开报告。

### 样式边界

- 使用 WPS 专属类名或 `data-provider-id="wps-export"` 范围选择器。
- 不改变其他 Provider 的 DOM 顺序、卡片样式和按钮行为。

## 进度日志设计

WPS 后端复用 `wandao_core.browser.emit` 和现有结构化日志协议，不修改通用日志解析器。

### 事件

1. `task.started`
   - 在取得待导出文件列表后立即输出。
   - 包含 `totals.documents` 和输出目录。

2. `document.export.started`
   - 每个文件开始处理前输出。
   - `doc` 包含文件 ID、标题、序号和目标路径。

3. `document.export.completed`
   - 文件成功写入或因已有文件而跳过后输出。
   - 包含本文件结果 `exported` 或 `skipped`。

4. `document.export.failed`
   - 单文件失败时输出，错误文本经过现有 `safe_error_text` 脱敏。
   - 失败后继续处理下一项。

5. `task.progress`
   - 每处理 `progress_every` 个文件输出一次，最后一项必须输出。
   - `progress.current` 表示已经结束处理的文件数量，`progress.total` 表示待处理总数。
   - `stats` 包含 `exportedDocs`、`skippedDocs`、`failureCount`。

### 进度规则

- 成功、跳过、失败都推进 current。
- checkpoint 已完成而被跳过的文件也推进进度。
- 日志不包含 Cookie、认证文件内容、下载 URL 或完整 API 响应。
- 用户停止时保留已经输出的最后进度；不为了进度功能修改其他平台的停止机制。

## 回退策略

- 不直接永久删除 `812603d` 中的全部 WPS UI 需求，因为该提交还包含 WPS 登录确认顺序要求。
- 以其父版本的全局结果布局为基准，撤销全局外置部分；随后只在 WPS Provider 内容内重新实现结果区。
- 最终 Git 差异必须能证明：除必要的通用渲染入口条件外，没有改变其他平台表现。

## 测试设计

### 前端回归测试

- 主页不包含 WPS 最近结果卡。
- 任务中心不包含 WPS 专属失败结果卡。
- 非 WPS Provider 不包含该卡。
- WPS Provider 内包含该卡。
- 卡片不存在“查看任务中心”和“打开报告”。
- WPS 登录确认按钮顺序保持现状。
- 失败项可展开，失败/部分完成默认展开。

### 后端回归测试

- 导出开始输出 `task.started`。
- 每个文件输出 started 和 completed/failed。
- 成功、跳过、失败均推进 `task.progress`。
- `progress_every` 生效，最后一项始终输出。
- 日志不泄露下载 URL、Cookie 和认证信息。
- 单文件失败不终止整个批次。

7. 本地提交并源码启动；不 push、不提 Issue、不提 PR。
