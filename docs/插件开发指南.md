# 万能导 Provider 共创指南

> 本文保留 Provider v1 的字段与兼容接入说明。需要提交可在插件中心按需安装、独立更新的正式平台插件，请优先阅读 [在线插件开发与发布](在线插件开发与发布.md)。Plugin v1 可以包含一个或多个 Provider v1。

万能导从 1.2.0 开始支持文件型 Provider。当前新增平台已经升级为 Plugin v1 主路径；本文主要说明旧文件型 Provider 的兼容规则，以及插件内部 Provider v1 entrypoint 的字段协议。

这套机制的目标是让社区接入新平台像“添加浏览器插件”一样清晰：平台能力放在自己的目录里，不影响其他平台；简单平台直接用标准 UI，复杂平台先提交核心脚本和流程说明，再评估是否升级成专属 UI。

新增平台请先阅读 [在线插件开发与发布](在线插件开发与发布.md)。只有维护旧 `providers/` 兼容目录时，才继续参考本文。

## Provider v1 兼容承诺

当前文件型 Provider 协议是 Provider v1，对应 `schemaVersion: 1`。从这一版开始，Provider v1 视为稳定契约。

你可以依赖这些稳定能力：

- `provider.json` 的基础字段：`id`、`name`、`title`、`description`、`type`、`group`、`trustLevel`、`status`、`capabilities`。
- 标准 UI 字段：`text`、`password`、`number`、`textarea`、`directory`、`file`、`checkbox`、`select`、`notice`。
- 动作按钮：`actions[].script`、`actions[].args`、`kind`、`includeSelection`、`updates`。
- 通用目录树：`toc` 映射和 `--doc-id` 这类选择参数。
- 脚本输出：普通日志、结构化日志、最终 JSON 报告。
- 运行时安全：脚本只能在 provider 自己目录内，社区 provider 执行前会提醒用户确认。

维护者后续可以增加可选字段，但不会在 v1 内删除已有字段或改变已有字段语义。如果未来确实需要破坏性升级，会新增 `schemaVersion: 2`，并保留 v1 兼容加载策略。

机器可读契约在：

```text
providers/provider.schema.json
```

## 5 分钟维护旧 Provider

推荐流程：

1. 先搜索已有 Issue/PR，确认没有重复共创。
2. 提交或认领“新平台共创/认领”Issue。
3. 根据平台类型复制 `providers/_template_standard/`、`providers/_template_import/` 或 `providers/_template_custom/`。
4. 把目录改名为你的平台 ID，例如 `providers/my-platform/`。
5. 修改 `provider.json` 里的 `id`、`name`、`title`、`description` 和 `capabilities`。
6. 修改 `fields`，声明用户需要填写的输入项。
7. 修改 `actions`，声明按钮要执行哪个 Python 脚本。
8. 在 `actions.py` 里实现读取目录、导出或导入。
9. 在 `README.md` 写清楚教程、登录方式、权限、限制和测试结果。
10. 本地运行检查后提交 PR。

以下划线开头的 provider 目录不会被自动加载，所以模板和示例都放在 `_template_*` 或 `_demo_*` 目录里。

## 选择哪种模板

标准导出模板：

```text
providers/_template_standard/
```

适合读取目录、勾选文档、导出 Markdown、复制图片和附件的平台。

标准导入模板：

```text
providers/_template_import/
```

适合扫描本地 Markdown、上传文档和资源、生成导入报告的平台。

自定义流程模板：

```text
providers/_template_custom/
```

适合复杂平台，例如多步骤登录、权限检测、空间选择、失败项重试、图片修复、断点续跑。当前文件型 provider 不直接注入任意 HTML；如果标准 UI 表达不了，请先提交 Python 核心逻辑、README 和 UI 需求说明，由维护者评估是否接入专属页面。

完整示例：

```text
providers/_demo_local_export/
```

这个示例会扫描本地 Markdown 目录，保留目录结构，并复制相对图片和附件。复制成 `providers/demo-local-export/` 后重启万能导即可加载测试。

## Provider 类型

- `automation`：只执行脚本。
- `guide`：只展示 `README.md` 教程，不执行脚本。
- `hybrid`：既展示教程，也提供脚本动作。

如果平台已经自带稳定导出功能，可以先做 `guide` 类型，把正确教程放进平台中心。

## provider.json 核心结构

```json
{
  "schemaVersion": 1,
  "id": "your-provider",
  "name": "你的平台",
  "title": "你的平台 Markdown 导出",
  "navLabel": "你的平台",
  "description": "一句话说明这个 provider 做什么",
  "type": "hybrid",
  "group": "export",
  "trustLevel": "community",
  "status": "experimental",
  "guide": "README.md",
  "capabilities": {
    "export": true,
    "import": false,
    "guide": true,
    "scanToc": true,
    "images": true,
    "attachments": false,
    "tree": true,
    "batch": true,
    "retryFailures": false
  },
  "retryFailures": {
    "arg": "--retry-failures",
    "label": "只重试失败项"
  },
  "fields": [],
  "actions": []
}
```

`id` 只能使用字母、数字、短横线和下划线，建议和目录名一致。

## 能力声明

`capabilities` 会告诉用户这个平台当前支持什么，也方便维护者判断 PR 范围：

- `export`：支持导出。
- `import`：支持导入。
- `guide`：有教程说明。
- `scanToc`：支持读取目录并勾选部分内容。
- `images`：支持图片处理。
- `attachments`：支持附件处理。
- `tree`：支持保留目录结构。
- `batch`：支持批量处理。
- `retryFailures`：支持根据上次报告只重试失败项。需要同时声明 `retryFailures.arg`，例如 `--retry-failures`；校验器会检查这个字段。

不要把还没实现的能力写成 `true`。可以先做最小可用版本，后续再补能力。

## 信任等级和状态

`trustLevel` 表示维护来源：

- `official`：官方内置维护。
- `community`：社区贡献，随项目发布。
- `local`：用户本机自行放入。
- `experimental`：实验性能力。

社区 PR 默认建议：

```json
{
  "trustLevel": "community",
  "status": "experimental"
}
```

`status` 可选值建议使用 `experimental`、`beta`、`stable`。真实测试较少时不要直接标成 `stable`。

## 依赖和权限说明

如果 provider 需要额外依赖，在 `requirements` 中声明：

```json
{
  "requirements": {
    "python": ["requests>=2.31.0", "beautifulsoup4>=4.12.0"],
    "system": ["Chrome 或 Edge"],
    "notes": ["首次使用需要登录目标平台。"]
  }
}
```

README 里还需要写清楚：

- 是否需要登录。
- 凭证保存在哪里。
- 凭证过期后用户会看到什么提示。
- 需要哪些平台权限。
- 是否调用官方 API、网页接口或浏览器自动化。
- 是否有频率限制或大批量导出风险。

## 标准 UI 字段

`fields` 用来声明表单，主程序会自动渲染，不需要贡献者写 Electron 前端。

支持字段类型：

- `text`
- `password`
- `number`
- `textarea`
- `directory`
- `file`
- `checkbox`
- `select`
- `notice`

示例：

```json
{
  "name": "output",
  "label": "输出目录",
  "type": "directory",
  "arg": "--output",
  "required": true
}
```

用户填写后，主程序会把它转换成命令行参数：

```text
--output 用户选择的目录
```

复选框示例：

```json
{
  "name": "incremental",
  "label": "跳过已存在文件",
  "type": "checkbox",
  "arg": "--incremental",
  "default": false,
  "advanced": true
}
```

`advanced: true` 会把字段放到高级参数区域。

字段默认会参与所有动作。为了让“登录 / 读取目录 / 导出 / 导入”各自只校验自己需要的字段，可以给字段加动作范围：

```json
{
  "name": "target_space",
  "label": "目标空间",
  "type": "select",
  "arg": "--target-space",
  "required": true,
  "actions": ["import"]
}
```

可用写法：

- `actions` / `includeActions`：只在指定动作中使用这个字段。
- `excludeActions` / `skipActions`：指定动作不使用这个字段。
- 没有额外声明时，`--output` / `output` 字段默认只参与 `export`、`import`、`run`、`start` 动作，不会阻塞 `scan`、`login`、`check` 这类准备动作。

## 动作按钮

`actions` 声明按钮：

```json
{
  "id": "export",
  "label": "开始导出",
  "script": "actions.py",
  "args": ["--export"],
  "actionName": "导出",
  "progressTitle": "平台导出",
  "progressDetail": "正在导出 Markdown..."
}
```

脚本路径只能指向 provider 目录内部的 `.py` 文件，不能使用 `../` 跳出目录。

常用字段：

- `kind: "scan"`：表示这是读取目录动作。
- `includeSelection: false`：不携带用户已勾选的目录项。
- `secondary: true`：渲染为次要按钮。
- `danger: true`：渲染为危险按钮。
- `confirm`：点击前弹出确认。
- `updates`：动作结束后把返回值回填到字段。

## 通用目录树协议

如果 provider 支持“读取目录 -> 勾选文档 -> 导出/导入选中项”，请声明 `toc`：

```json
{
  "capabilities": {
    "scanToc": true
  },
  "toc": {
    "itemsPath": "nodes",
    "idKey": "nodeId",
    "exportIdKey": "exportId",
    "titleKey": "title",
    "parentIdKey": "parentNodeId",
    "selectableKey": "selectable",
    "selectionArg": "--doc-id"
  }
}
```

读取目录动作返回：

```json
{
  "provider": "your-provider",
  "nodes": [
    {
      "nodeId": "folder:docs",
      "exportId": "",
      "title": "docs",
      "parentNodeId": "",
      "selectable": false
    },
    {
      "nodeId": "doc:docs/hello.md",
      "exportId": "docs/hello.md",
      "title": "hello",
      "parentNodeId": "folder:docs",
      "selectable": true
    }
  ]
}
```

用户勾选后，后续动作会自动追加：

```text
--doc-id docs/hello.md
```

## 动态字段回填

有些平台需要先读取空间、知识库或文件夹，再填充下拉框。可以用 `actions.updates`：

```json
{
  "id": "list-spaces",
  "label": "读取空间列表",
  "script": "actions.py",
  "args": ["--list-spaces"],
  "includeSelection": false,
  "updates": [
    {
      "field": "target_space",
      "type": "options",
      "path": "spaces",
      "valueKey": "id",
      "labelKey": "name",
      "placeholder": "请选择目标空间"
    }
  ]
}
```

脚本返回：

```json
{
  "spaces": [
    {"id": "space_1", "name": "默认空间"}
  ]
}
```

UI 会自动把 `target_space` 下拉框更新为返回的选项。

## 脚本输出协议

脚本可以持续输出普通日志。为了让任务中心和错误报告能定位问题，推荐优先使用结构化日志。

最简单的方式是在脚本里引用项目内置的 `wandao_logging.py`：

```python
from wandao_logging import WandaoLogger

log = WandaoLogger(provider="your-provider")

log.info("task.started", "开始导出", totals={"documents": 10})
log.progress(3, 10, event="task.progress", stats={"exportedDocs": 3, "failureCount": 0})
log.error(
    "resource.download.failed",
    "图片下载失败",
    resource={"type": "image", "url": "https://example.com/a.png"},
    error="HTTP 403",
)
log.info("task.completed", "导出完成", stats={"exportedDocs": 10, "failureCount": 0})
```

文件型 provider 也可以直接引用 `wandao_logging.py`。桌面端会自动把万能导内置 Python 公共目录加入 `PYTHONPATH`，打包后仍然可用。

推荐事件名：

```text
task.started / task.progress / task.completed / task.failed / task.stopped
auth.started / auth.saved / auth.expired
toc.scan.started / toc.scan.completed / toc.scan.failed
document.export.started / document.export.completed / document.export.failed
document.import.started / document.import.completed / document.import.failed
resource.download.started / resource.download.completed / resource.download.failed
resource.upload.started / resource.upload.completed / resource.upload.failed
api.request / api.response / api.failed
browser.started / browser.connected / browser.failed
report.written
```

如果暂时不接结构化日志，也可以继续输出普通文本。任务结束时，最后仍要输出一段普通 JSON，便于万能导记录任务历史。注意：最终结果 JSON 不要加 `@@WANDAO_LOG@@` 前缀，也不要只通过 `log.info(...)` 输出。

```json
{
  "provider": "your-provider",
  "mode": "export",
  "totalDocs": 10,
  "exportedDocs": 10,
  "skippedDocs": 0,
  "failureCount": 0,
  "failures": []
}
```

普通文本进度日志仍然兼容：

```text
progress 3/10 exported=3 skipped=0 failures=0
```

失败项建议包含文档 ID、标题和错误原因：

```json
{
  "failures": [
    {
      "docId": "abc123",
      "title": "失败文档",
      "error": "图片上传超时"
    }
  ]
}
```

## 复杂平台怎么共创

复杂平台不要一开始就大改主程序。建议分三步：

1. 先用 `README.md` 写清楚人工流程和平台限制。
2. 再用 Python 脚本跑通核心导入导出能力。
3. 如果标准 UI 不够，在 PR 中说明需要的专属 UI、按钮、字段和状态流转。

例如飞书导入这类功能，完全可以共创，但要把“权限检测、读取目标空间、图片补全、导入、失败重试、报告”拆清楚。维护者可以基于你的核心脚本接入专属 UI，而不会影响其他平台。

## AI 辅助开发提示词

新平台请使用 Plugin v1 提示词：

```text
docs/tutorials/provider-ai-prompt.md
```

只有维护旧 `providers/` 兼容目录时，才参考本文和 `providers/_template_*`。

## 提交 PR 前检查

- `provider.json` 是合法 JSON。
- `id` 与目录名一致，且不和现有 provider 冲突。
- `README.md` 写明使用步骤、限制、登录方式、权限和测试结果。
- `capabilities` 没有声明尚未实现的能力。
- Python 脚本可以通过 `python -m py_compile`。
- 没有提交 Cookie、Token、账号密码、App Secret 或私人测试数据。
- 如果参考第三方项目，已说明来源和许可证。

建议提交前运行：

```powershell
python scripts\validate_providers.py
python scripts\quality_check.py
```

## 合规要求

万能导采用 AGPL-3.0-only 协议开源。提交 provider、脚本或文档时，默认表示你同意按本项目协议授权贡献内容。

万能导只用于整理用户自己有权限访问的内容。不要提交绕过登录、绕过权限校验、破解平台限制或批量抓取未授权内容的实现。
